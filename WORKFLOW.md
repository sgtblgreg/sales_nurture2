# Sales Nurture — рабочий процесс (локально → релиз)

Этот репозиторий — **источник истины**. GitHub Pages публикует содержимое репозитория (обычно ветку `main`).
Отдельная “папка на GitHub” не нужна: GitHub‑версия = то, что **запушено** в `origin`.

## Базовый цикл (рекомендуется)
1) Обновить локальную копию:
   - `git checkout main`
   - `git pull --ff-only`
2) Создать рабочую ветку:
   - `git checkout -b feature/<коротко-что-делаем>`
3) Внести изменения.
4) Протестировать локально **через HTTP**, не `file://`:
   - `python3 -m http.server 8000`
   - открыть `http://localhost:8000/`
   - DevTools → Network → убедиться, что нет `404` по Img/Doc (лого, PDF, превью).
5) Зафиксировать изменения:
   - `git add -A`
   - `git commit -m \"...\"`
6) Релиз (ты делаешь сам):
   - вариант A (просто): `git push origin feature/<...>` → Merge (PR) → Pages обновится
   - вариант B (локальный merge): `git checkout main` → `git merge --no-ff feature/<...>` → `git push origin main`

## Почему “дубли” не нужны
Если ты хочешь “оставить старую версию на всякий случай” — это уже есть в git‑истории.
Для экспериментов лучше:
- ветка `wip/<...>` (не мерджить в `main`);
- или тег `backup/<date>` перед крупными правками.

## Правила для ассетов (важно для GitHub Pages)
- **Не используем `<base>`**: пути должны быть стабильными без него.
- Для runtime‑ссылок в `index.html` используется `assetUrl(...)`, который учитывает project pages (`/<repo>/`).
- GitHub Pages по умолчанию прогоняет сайт через Jekyll и **не публикует** директории, начинающиеся с `_`.
  Поэтому для превью в `Example researches/_thumbs/` в корне репозитория должен быть `.nojekyll`.

## Превью PDF (без 404)
- Превью показываем **только** если путь явно добавлен в `sales_nurture_data.js` → `asset_manifest.thumbs`.
- Если превью не указано — UI показывает placeholder и **не делает сетевой запрос**.
